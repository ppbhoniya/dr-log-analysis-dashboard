# DR Log Analysis Dashboard - Technical Specification

**Version:** 1.0  
**Date:** April 2026  
**Stack:** Python 3.x + Flask · Vanilla HTML / CSS / JavaScript  

---

## 1. Purpose

The application provides a real-time, browser-based monitoring tool for operations engineers managing protection relay systems.

It correlates **Disturbance Record (DR) files** from three distinct industrial log sources:

1. **FTP Log** — generated by the OTI FTP polling service
2. **OTI Log** — generated by the ETAP Notifier service (`etap-notifier-*.log`)
3. **FaultLocation Log** — generated by the ETAP AFAS/FaultLocation service (`FaultLocation-*.log`)

The goal is to identify DRs that have been captured by the FTP system but have **not yet been processed** by the OTI or FaultLocation services, so engineers can investigate missing events quickly.

---

## 2. Architecture

```
Browser (Vanilla JS)
    |
    |  HTTP POST /api/data (every 60s)
    v
Flask Server (app.py)
    |
    |-- parse_ftp(ftp_path)       → Dict[dr_name, {ip, device_id}]
    |-- parse_oti(oti_path)       → Dict[dr_name, {ip}]
    |-- parse_fl(fl_folder_path)  → Dict[dr_name, {device_id (relay)}]
    |
    └── Match Engine → HTTP JSON Response → JS renders table
```

## 3. Backend Specification (`app.py`)

### 3.1 FTP Log Parser — `parse_ftp(filepath)`

**Input:** Path to FTP application log file (e.g., `FTP.exe_1`)

**Key Patterns:**

| Pattern | Purpose |
|---------|---------|
| `Calling ListDirectory for 'DEVICE_ID' on path: ftp://IP:21/` | Maps IP → Device ID |
| `From path: ftp://IP:21/` | Sets current active IP context |
| `drec_NNN.zip` (in file listing block) | Extracts DR file name |

**Exclusions:** Files matching `drec_NNNh.zip` are excluded (header-only files).

**Output:** `{ "drec_NNN": { "ip": "172.x.x.x", "device_id": "NS186021P02_1" } }`

---

### 3.2 OTI Log Parser — `parse_oti(filepath)`

**Input:** Path to `etap-notifier-*.log`

**Key Pattern:**

| Pattern | Purpose |
|---------|---------|
| `ftp://IP:21/.../drec_NNN` (any line) | Extracts DR file name and IP |

**Exclusions:** DR names ending in `h` are excluded.

**Output:** `{ "drec_NNN": { "ip": "172.x.x.x" } }`

---

### 3.3 FaultLocation Log Parser — `parse_fl(folderpath)`

**Input:** Path to a folder (iterates all `.log` files) OR a single `.log` file path.

**Key Pattern:**

| Pattern | Purpose |
|---------|---------|
| `AFAS: NEW DR DETECTED ... relay=RELAY_NAME ... cfg=drec_NNN.cfg` | Extracts DR file name and relay name |

**Exclusions:** DR names ending in `h` are excluded.

**Output:** `{ "drec_NNN": { "device_id": "R1_NS-186021B" } }`

---

### 3.4 Matching Engine

All unique DR names from all three parsers are collected into a set.  
For each DR, the following `status` is assigned:

| Condition | Status |
|-----------|--------|
| Present in FTP + OTI + FaultLocation | `Matched in All` |
| Present in FTP, missing in both OTI and FaultLocation | `Missing in Both` |
| Present in FTP, missing only in OTI | `Missing in OTI` |
| Present in FTP, missing only in FaultLocation | `Missing in FaultLocation` |
| Missing from FTP, present in OTI/FL | `Missing in FTP` |

---

### 3.5 API Endpoint

**`POST /api/data`**

Request body:
```json
{
  "ftp_path": "D:\\Logs\\FTP.exe_1",
  "oti_path": "D:\\Logs\\etap-notifier.log",
  "fl_path": "D:\\Logs\\FaultLocation\\"
}
```

Response:
```json
{
  "success": true,
  "data": [
    {
      "dr_name": "drec_293",
      "ip": "172.17.201.2",
      "device_id": "NS186021P02_1",
      "fl_relay": "R1_NS-186021B",
      "in_ftp": true,
      "in_oti": true,
      "in_fl": false,
      "status": "Missing in FaultLocation"
    }
  ]
}
```

## 4. Frontend Specification

### 4.1 Input Section

Three text inputs for file/folder paths with pre-filled defaults. A single **Start / Stop Polling** button toggles a 60-second `setInterval` polling loop.

### 4.2 Summary Cards

Three animated counter cards update on each poll:

- **Total DR Files Processed** (purple)
- **Matched in All** (green)
- **Missing / Errors** (red)

Values animate with a number-counting animation over 600ms.

### 4.3 Log Details Table

Grouped by **Device ID**. Columns:

| Column | Data Source |
|--------|-------------|
| Device ID + ⚡ FL Relay | `device_id` from FTP, `fl_relay` from FaultLocation |
| IP Address | From FTP or OTI |
| Total DRs | Count of unique DRs for this device |
| FTP DRs | DR names found in FTP; green if matched, red if missing in other source(s) |
| OTI DRs | DR names found in OTI; color matches match status |
| FaultLocation DRs | DR names found in FL (green) OR expected but missing (red). Hover = relay name tooltip |
| Matched | Count of fully matched DRs |
| Status | Descriptive badge |

### 4.4 Color Conventions

| Color | Hex / Variable | Meaning |
|-------|---------------|---------|
| Green | `var(--green-success)` `#10b981` | Matched in all sources |
| Red | `var(--red-alert)` `#ef4444` | Missing/error |
| Purple | `#a78bfa` | FaultLocation relay name |
| Red row | `rgba(239,68,68,0.1)` border-left | Device has at least one mismatch |

### 4.5 Status Badge Text

| Status | Badge Color |
|--------|------------|
| `All Good` | Green |
| `Missing in FaultLocation` | Red |
| `Missing in OTI` | Red |
| `Missing in FTP` | Red |
| `Multiple Errors/Missing` | Red |

---

## 5. Polling & Refresh

- Frontend uses `setInterval(fetchData, 60000)` — fires every **60 seconds**.
- On each poll, a `POST /api/data` request is made.
- The backend reads files **fresh from disk** on every request (no caching).
- The table is fully re-rendered from the latest response on each poll.
- The "Last updated" timestamp is shown in the table header.

## 6. Deployment

### Local Run

```bash
cd dashboard
pip install -r requirements.txt
python app.py
```

Open `http://127.0.0.1:5000`

### Production Notes

- Replace `app.run(debug=True)` with a WSGI server (e.g., `gunicorn app:app`)
- All log files must be accessible from the server's filesystem path
- Suitable for deployment on a local network server accessible by operations staff
